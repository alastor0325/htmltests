Seek-to-first-frame latency — Bug 2049301

Measures video.currentTime = target latency for a buffered ±10 s seek, across three codecs (H.264 / VP9 / AV1) and two playback states:

  • Playback seek — the video is playing when the seek is issued, so the platform decoder is alive (flushed, not torn down).
  • Dormant seek — the video is paused long enough to go dormant (Firefox releases the decoder after media.dormant-on-pause-timeout-ms, default 5 s), so each seek recreates the decoder. The wait below is set just above that timeout. Other browsers have no dormancy concept, so for them this is simply a paused seek.

Primary metric is the seeked event (decode reached the target frame); firstFrame is the requestVideoFrameCallback time (best-effort, presentation).
